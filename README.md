# CFG Eval Toy

Natural language queries against a ClickHouse `orders` table, with SQL
generated by GPT-5 through a context free grammar constraint. Ships with
22 evals across three categories and a reliability panel for running cases
multiple times.

## What it does

Type a natural language question about the orders table into the query box.
GPT-5 generates a ClickHouse SQL query constrained by a hand written Lark
grammar, so the output is guaranteed parseable, scoped to one table, and
limited to a specific set of SELECT operations. The SQL runs against
ClickHouse Cloud and the results come back.

The reliability panel below the query box runs any canonical eval case up
to 5 times and shows the raw pass count plus the SQL that came back each
time. This is Mode A (stochasticity): same input, repeated.

## Dataset

Synthetic 10,000 row `orders` table in ClickHouse Cloud, generated with a fixed seed for reproducibility.

**Schema:**

| Column | Type | Example |
|---|---|---|
| order_id | String (UUID) | `40d6976c-45f2-4f50-87b0-8fd051ac756e` |

## Stack

- **Next.js 15 + React 19 + TypeScript**, deployed on Vercel
- **OpenAI Responses API** with the CFG tool (Lark grammar, ~50 rules)
- **ClickHouse Cloud** (free tier)
- **Synthetic 10k row dataset** with pinned `NOW()` for deterministic evals

## Evals

22 cases across three categories:

| Category | Count | What they test |
|---|---|---|
| Canonical | 6 | One per semantic intent: sum, count, group by avg, max, numeric predicate, group by count. Each asserts an exact scalar or row count using pinned `NOW()`. |
| Paraphrases | 13 | 2-3 per canonical intent. Same semantic, different wording. Tests the model's robustness to rephrasing. |
| Edge cases | 3 | SQL injection attempt, out of scope query, unbounded aggregate. Shows what the CFG catches. |

Each case uses one or more of:

- **Value assertions**: `expected_scalar` or `expected_row_count` with tolerance, computed against pinned `NOW()`
- **Stage assertions**: `expected_stage: [grammar_fail, ok]` for cases where multiple outcomes are safe

Run the canonical suite:

```bash
npx tsx --env-file=.env.local scripts/verify-baseline.ts
```

Run the full 22 case suite:

```bash
npx tsx --env-file=.env.local scripts/verify-all.ts
```

Or use the reliability panel in the deployed UI.

## Exploration pass

Before recording the walkthrough I ran an offline exploration pass, every
case N times, grouped by exact raw SQL, with Wilson 95% CIs:

- canonical: **10 trials** each
- paraphrase: 5 trials each
- edge: 5 trials each
- **140 trials total, ~19 min wall clock, all serial**

Full log: [`evals/exploration-pass-2026-04-08.txt`](evals/exploration-pass-2026-04-08.txt)

**140/140 trials passed. 22/22 cases at 100% pass rate. 10/22 cases
produced multiple SQL variants.** The model picked different syntactic forms
across repeated trials on the same prompt, inside the same CFG constraint.
Every variant observed in this run produced the same answer, which is easy
to read as "no big deal." But that's just what 140 trials at n=5 or n=10
happened to sample. The interesting question is not whether the eval passed
but *how many ways* it passed.

The sharpest example: `count_over_200_30d_canonical` split 5/5 across
two WHERE clause orderings.

```sql
-- 5 of 10 trials:
SELECT count(*) FROM orders
WHERE price > 200 AND order_purchase_timestamp >= now() - INTERVAL 30 DAY

-- 5 of 10 trials:
SELECT count(*) FROM orders
WHERE order_purchase_timestamp >= now() - INTERVAL 30 DAY AND price > 200
```

Clean coin flip, same answer, two stable forms the model sampled from
inside the grammar. The reliability panel in the UI reproduces this live.

Observed drift kinds (all semantically equivalent in this run):

| Kind | Example | Cases affected |
|---|---|---|
| Predicate order flip | `A AND B` vs `B AND A` | 6 |
| SELECT column order | `avg(price), state` vs `state, avg(price)` | 2 |
| Interval literal choice | `1 WEEK` vs `7 DAY`, `1 MONTH` vs `30 DAY` | 2 |
| Refusal vs safe fallback | `grammar_fail` vs benign `SELECT count(*)` | 1 (SQL injection) |

### When drift changes the answer

During eval development I hit one case where the drift *did* change the
answer. The `count_over_200_30d_para2` paraphrase ("how many high value
orders were placed in the last month") has GPT-5 choosing between
`INTERVAL 30 DAY` and `INTERVAL 1 MONTH`. The grammar admits both. On this
dataset's calendar boundary, `30 DAY` returns 65 rows and `1 MONTH` returns
67 because calendar months are ~30.4 days, not exactly 30. Two grammatically
legal productions, different semantics, different row counts.

I widened `expected_scalar` to `66 +/- 2` to accept both outcomes and left a
comment in [`evals/cases.yaml`](evals/cases.yaml) explaining the tolerance.
The exploration pass didn't reproduce this drift (all 5 trials picked
`INTERVAL 1 MONTH`), which is the other thing the panel makes visible:
small samples hide rare drift, and "100% pass at n=5" is a weaker claim
than it looks.

The finding:

> **CFG correctness is necessary but not sufficient.** The grammar can
> admit two equally legal productions whose semantics differ on the actual
> data. The only way to see that is to pin `NOW()` and assert on the
> executed result, not on the generated string.

Pinned `NOW()` plus exact scalar assertions plus multi trial sampling is
textbook execution accuracy. The contribution here is the panel: it
surfaces the distribution of generated SQL (how many ways did this case
pass?) as a signal next to the pass rate.

## Running locally

1. Clone and install:
   ```bash
   npm install
   ```

2. Create `.env.local` with:
   ```
   OPENAI_API_KEY=sk-...
   CLICKHOUSE_URL=https://...clickhouse.cloud:8443
   CLICKHOUSE_USER=default
   CLICKHOUSE_PASSWORD=...
   ```

3. Generate and ingest the synthetic dataset:
   ```bash
   python scripts/generate-dataset.py
   npm run ingest
   ```

4. Start the dev server:
   ```bash
   npm run dev
   ```

## Architecture

- **`lib/pipeline.ts`** is the single shared path: both `/api/query` (live
  queries from the UI) and `lib/eval-runner.ts` (eval trials) call the same
  `runPipeline(nl)` function. The reliability panel measures exactly the
  code path the user's query takes.
- **`lib/sanitize-error.ts`** sits between the pipeline and the API route.
  Classifies errors into short user safe messages and logs the full stack
  server side.
- **`grammar/orders.lark`** is the CFG. ~50 rules, explicit `SP`/`COMMA`
  terminals for whitespace control. Only admits SELECTs against the `orders`
  table with enumerated columns and a restricted set of predicates.
  `DROP`/`DELETE`/`INSERT`/`JOIN` are grammatically inexpressible.
- **`lib/pinned-now.ts`** caches `max(order_purchase_timestamp)` on first call
  and rewrites `now()` in generated SQL to that literal before execution.
  This makes "last 30 hours" queries return the same answer today as they
  will next month.
